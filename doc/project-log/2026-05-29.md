## 背景问题

在第一优先级完成后，Codex 会话连续性和重复消息问题已经修复并发布。但默认执行策略仍然偏激进：

- 后端默认使用 `DangerFullAccess`。
- 后端默认开启网络访问。
- 当前应用尚未实现交互式审批 UI，因此不能直接依赖 `OnRequest` 审批模式。

这对真实仓库使用风险较高，尤其是执行任务时可能修改工作区外文件或运行联网命令。

## 本次改动

### 1. 安全配置模型

前端 `AgentConfig` 和 `AgentRequest` 新增：

- `sandboxMode`
- `networkAccessEnabled`

其中 `sandboxMode` 支持：

- `read-only`
- `workspace-write`
- `danger-full-access`

### 2. 默认权限收紧

默认 Codex Agent 配置改为：

- `sandboxMode: workspace-write`
- `networkAccessEnabled: false`

这意味着默认只允许在工作区范围内写入，并关闭生成命令的 shell 网络访问。

### 3. 设置界面新增 Execution Safety

在 Codex CLI Agent 配置编辑器中新增 `Execution Safety` 区域：

- 可选择 `Read only`
- 可选择 `Workspace write (recommended)`
- 可选择 `Full access`
- 可切换 `Shell network access`

当用户选择 `Full access` 时，界面会显示明显风险提示，并将网络访问视为开启状态，因为全权限模式本身移除了文件系统边界。

### 4. Rust 执行策略映射

Rust 后端根据请求中的 `sandboxMode` 映射到 Codex SDK：

- `read-only` -> `SandboxMode::ReadOnly`
- `workspace-write` -> `SandboxMode::WorkspaceWrite`
- `danger-full-access` -> `SandboxMode::DangerFullAccess`

如果旧配置没有传入沙盒模式，则默认使用 `workspace-write`。

### 5. 保留审批策略现状

当前仍保留：

```rust
ApprovalMode::Never
```

原因是应用还没有实现 Codex 审批事件到用户确认 UI 的完整闭环。直接切换到请求审批模式可能导致任务运行卡住或无法继续。
