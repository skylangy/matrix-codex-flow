# Matrix Codex Flow 项目记录 - 2026-05-24

## 今日概览

本日完成项目结构梳理，并修复 Codex 聊天主链路中的会话恢复与重复消息问题。当前项目是基于 Angular 21 与 Tauri 2 的本地桌面应用，通过 Rust 服务连接 Codex SDK，以 SQLite 保存项目、任务与聊天记录。

## 项目结构速览

| 模块 | 技术/目录 | 作用 |
| --- | --- | --- |
| 前端界面 | `src/app`、Angular 21 | 欢迎页、工作区、聊天、上下文规则、任务管理、设置 |
| 桌面运行时 | `src-tauri`、Tauri 2 | 窗口生命周期、前后端命令桥接 |
| 数据持久化 | Rust + SQLite | 保存项目、规则、任务、聊天线程和消息 |
| Agent 执行 | `codex-sdk-rs` | 启动或恢复 Codex thread，并转发事件 |
| 项目上下文 | `.codex/*.md` | 导出规则和任务文档，供项目内复用 |

## 今日完成事项

### 1. 项目分析

- 确认实际技术栈为 Angular 21 + Tauri 2 + Rust + SQLite。
- 梳理主要页面路径：`home`、`workspace/chat`、`workspace/context`、`workspace/tasks` 和 `settings`。
- 梳理调用链：前端 Provider -> Tauri `invoke` -> Rust `CodexService` -> Codex SDK -> Tauri 事件 -> 前端消息存储。
- 发现 README 中 Electron 描述与实际 Tauri 架构不一致，后续应更新文档。

### 2. Codex 事件协议修复

问题：

- 后端将 thread 开始事件编码为 `type/data` 结构，前端却直接读取顶层 `thread_id`，导致 Codex thread id 无法正确保存。
- 前端尝试从 `done` 事件读取正文，但后端完成事件只提供 token 数量。

处理：

- 统一前端事件类型定义，正确解析 `threadStarted.data.threadId` 和 `done.data.totalTokens`。
- 将错误事件转为失败结果，使调用方能够识别 Codex 执行失败。

### 3. 会话恢复能力补齐

问题：

- 本地聊天线程没有保存其对应的 Codex thread id，切换历史线程或重启应用后无法继续原有 Agent 会话。

处理：

- 为聊天线程新增 `agentThreadId` 字段。
- 在新建 Codex thread 后保存 thread id，并在下一次请求中继续携带。
- 为 SQLite 的 `chat_threads` 表增加 `agent_thread_id` 列及兼容旧数据库的自动迁移逻辑。

### 4. 重复消息入库修复

问题：

- Rust 同时向前端推送 `ItemUpdated` 和 `ItemCompleted`，前端又把每次回调保存为独立消息，可能造成同一次回答被重复记录。

处理：

- 后端只将完成态 item 发送给前端作为可持久化消息。
- 前端在单次响应内使用同一消息 ID 更新内容，保存采用 upsert 逻辑。
- 保留 Markdown 与代码正文中的原始空白，避免合并响应时损伤格式。
